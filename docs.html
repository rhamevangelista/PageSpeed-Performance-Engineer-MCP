Getting Started

Installation

Requires Node.js 22+. Playwright downloads Chromium automatically on install.

1
Clone and install
bash
git clone https://github.com/your-repo/performance-mcp
cd performance-mcp
npm install    # playwright install chromium runs automatically
2
Set up environment variables (optional)
bash
cp .env.example .env
# Add GOOGLE_PAGESPEED_API_KEY for real-user CrUX field data
i
Without an API key, get_core_web_vitals returns Lighthouse lab simulation. With a key it returns real-user CrUX field data from PageSpeed Insights.
3
Build the server
bash
npm run build   # compiles TypeScript to dist/server.js
npm start       # verify: should print "MCP server running on stdio"
Configuration

Connecting to Claude

Add the server to your MCP configuration. The path must point to the compiled dist/server.js.

Claude Code (VS Code / CLI)

Edit .claude/settings.json in your project, or the global ~/.claude/settings.json:

json.claude/settings.json
{
  "mcpServers": {
    "performance": {
      "command": "node",
      "args": ["C:/path/to/performance-mcp/dist/server.js"]
    }
  }
}

Claude Desktop

Edit claude_desktop_config.json (found in Claude Desktop settings → Developer):

jsonclaude_desktop_config.json
{
  "mcpServers": {
    "performance": {
      "command": "node",
      "args": ["/Users/you/performance-mcp/dist/server.js"]
    }
  }
}

Restart Claude after saving. The “performance” server will appear in available MCP servers.

Core Concepts

Core Web Vitals thresholds

The five metrics this tool measures and optimizes for. All tools report in these units.

LCP
Largest Contentful Paint
≤ 2.5s good
≤ 4.0s needs work
> 4.0s poor
CLS
Cumulative Layout Shift
≤ 0.1 good
≤ 0.25 needs work
> 0.25 poor
INP
Interaction to Next Paint
≤ 200ms good
≤ 500ms needs work
> 500ms poor
FCP
First Contentful Paint
≤ 1.8s good
≤ 3.0s needs work
> 3.0s poor
TTFB
Time to First Byte
≤ 800ms good
≤ 1.8s needs work
> 1.8s poor
Tools Reference

All 16 tools

Blue badges are required parameters. All tools return deterministic JSON — no prose.

Phase 1 — CoreFoundation
analyze_website
Full orchestrator. Auto-detects CMS, fans out all tools in parallel, dispatches platform-specific inspection, and returns a prioritized recommendation list. Start here for any audit.
url: string
run_lighthouse
Runs a Lighthouse audit via the npm package. Returns performance score, all 5 Core Web Vitals (lab data), and improvement opportunities with estimated time/byte savings.
url: stringstrategy: mobile|desktop
get_core_web_vitals
Fetches from PageSpeed Insights API. Returns real-user CrUX field data when an API key is set, otherwise falls back to Lighthouse lab data.
url: stringstrategy: mobile|desktop
analyze_headers
Inspects HTTP response headers: Brotli/gzip compression, Cache-Control, ETag, HTTP/2 support, CDN detection, and HSTS/CSP/X-Frame security headers.
url: string
analyze_html
Parses HTML with Cheerio. Checks render-blocking stylesheets and scripts, defer/async usage, inline script count, preload hints, canonical tag, and viewport.
url: string
detect_cms
Fingerprints the CMS and hosting platform using HTML patterns, HTTP headers, and URL conventions. Identifies WordPress, Shopify, Next.js, Nuxt, Astro, Webflow, Wix, Squarespace, Drupal, Joomla, and more.
url: string
Phase 2 — Deep AnalyzersPlaywright
analyze_images
Scans every img element via Playwright. Detects oversized images, legacy formats (PNG/JPEG vs AVIF/WebP), missing lazy loading, missing width/height attributes, and identifies the LCP hero candidate to preload.
url: string
analyze_css
Uses Playwright CSS coverage to measure per-stylesheet usage %. Flags render-blocking stylesheets, high unused CSS, and large framework bundles (Bootstrap, Tailwind full build).
url: string
analyze_javascript
Uses Playwright JS coverage to find unused JS per bundle. Detects render-blocking scripts, large bundles, and 15+ third-party vendors (GTM, Facebook Pixel, Hotjar, Intercom, Segment, etc.).
url: string
analyze_fonts
Checks Google Fonts preconnect hints, CSS @import anti-pattern, font-display values, WOFF2 format usage, and preload hints for self-hosted fonts.
url: string
Phase 3 — Platform InspectorsCMS-Aware
wordpress_inspector
Deep WordPress audit. Detects 20+ plugins (Elementor, Divi, WooCommerce, WP Rocket, LiteSpeed Cache, NitroPack, Perfmatters, FlyingPress, Yoast, RankMath, and more) with plugin-level recommendations and estimated LCP impact.
url: string
cloudflare_inspector
Detects Cloudflare and audits CF-Cache-Status, Brotli compression, HTTP/3, Early Hints, and Polish image optimization. Returns structured Cache Rules, Transform Rules, and dashboard setting recommendations.
url: string
Phase 4 — Intelligence LayerGenerators
generate_report
Runs a full analysis and returns a human-readable Markdown report: Lighthouse score tables, CWV with good/warn/poor ratings, platform summary, image/JS/CSS summaries, and numbered fix recommendations.
url: string
generate_wp_rocket_config
Analyzes WordPress plugins and returns a WP Rocket settings JSON object. Accounts for Elementor/Divi (disables CSS/JS combination), WooCommerce cart exclusions, script delay rules, and conflict warnings for NitroPack and LiteSpeed.
url: string
generate_cloudflare_rules
Generates ready-to-implement Cloudflare Cache Rules (static assets 1yr, HTML 5min), Transform Rules (security headers), Speed settings, and CMS-aware cache bypass rules for WordPress admin, WooCommerce cart, and Shopify checkout.
url: string
compare_performance
Compares two analyze_website JSON reports (before and after a change). Returns Lighthouse score deltas, CWV improvements or regressions, newly introduced issues, and resolved issues with a summary sentence.
before_report: JSON stringafter_report: JSON string
Usage

Common workflows

Full site audit — start here

analyze_website detects the CMS and fans out all tools in parallel. It returns everything in one call.

claude
analyze_website { "url": "https://example.com" }

Get a shareable Markdown report

claude
generate_report { "url": "https://example.com" }

WordPress + WP Rocket workflow

Step 1
wordpress_inspector
Detect active plugins, page builder, WooCommerce, and existing cache plugins
Step 2
generate_wp_rocket_config
Generate recommended WP Rocket settings JSON tailored to what was detected
Step 3
analyze_images
Find images to convert to AVIF/WebP and identify the LCP preload candidate
claude
# Deep WordPress plugin audit
wordpress_inspector { "url": "https://my-wp-site.com" }

# Generate WP Rocket config tailored to this site
generate_wp_rocket_config { "url": "https://my-wp-site.com" }

Cloudflare setup workflow

claude
# Audit current Cloudflare status
cloudflare_inspector { "url": "https://example.com" }

# Generate Cache Rules + Transform Rules to implement
generate_cloudflare_rules { "url": "https://example.com" }

The rules output contains the exact Cloudflare expression syntax ready to paste into the Cache Rules editor in the Cloudflare dashboard.

Before/after comparison

Run analyze_website before making changes, save the JSON, make improvements, run again, then compare the two outputs.

claude
# Step 1: run before changes, copy the JSON output
analyze_website { "url": "https://example.com" }

# Step 2: make your changes, run again, copy new JSON
analyze_website { "url": "https://example.com" }

# Step 3: compare the two reports
compare_performance {
  "before_report": "{ ...first JSON... }",
  "after_report": "{ ...second JSON... }"
}
⚠
Lighthouse lab scores vary run-to-run by ±5 points. For stable comparisons, use get_core_web_vitals with a PageSpeed API key to get real-user CrUX field data instead.

Targeted single-tool audits

claude
# Images only
analyze_images { "url": "https://example.com" }

# JavaScript bundles and third-party scripts only
analyze_javascript { "url": "https://example.com" }

# Font loading strategy only
analyze_fonts { "url": "https://example.com" }

# Desktop Lighthouse audit
run_lighthouse { "url": "https://example.com", "strategy": "desktop" }
